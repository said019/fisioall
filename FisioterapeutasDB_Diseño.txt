1. Introducción y Objetivos
Este documento define la arquitectura completa de base de datos para una plataforma SaaS multi-tenant dirigida a fisioterapeutas y clínicas de rehabilitación. El sistema cubre el ciclo de vida completo del paciente: desde el diagnóstico inicial hasta el alta, incluyendo agenda, membresías, expediente clínico digital, seguimiento y pagos.

1.1 Principios de Diseño
	•	Multi-tenancy nativa: aislamiento por tenant_id + Row Level Security (RLS) de PostgreSQL
	•	UUIDs como PK: portabilidad entre entornos y seguridad frente a enumeración
	•	Datos clínicos normalizados: SOAP, body map, escala EVA de dolor como columnas tipadas
	•	Automatización via triggers: sesiones, progreso, timestamps — sin lógica en la app
	•	Performance: índices GIN para búsquedas fuzzy, GIST para detección de solapamiento de horarios
	•	Extensibilidad: columnas JSONB para configuración y atributos variables sin ALTER TABLE

1.2 Stack Tecnológico
	•	Motor: PostgreSQL 15 + extensiones uuid-ossp, pgcrypto, pg_trgm
	•	ORM sugerido: Prisma o Drizzle (compatible con tu stack React + Node.js + Firebase Auth)
	•	Hosting: Supabase (managed PostgreSQL con RLS nativo y edge functions) o Railway
	•	Backups: pg_dump diario + WAL-G para point-in-time recovery

5. Triggers y Funciones PostgreSQL
La automatización a nivel de base de datos elimina lógica de negocio crítica de la capa de aplicación, garantizando consistencia incluso ante accesos directos o errores en el código de la app.

Trigger
Evento
Descripción

Actualiza el campo updated_at automáticamente en cualquier modificación

5.1 Función: verificar_disponibilidad()
Función utilitaria que verifica si un fisioterapeuta tiene disponibilidad en un rango de tiempo dado, usando el operador && sobre tstzrange indexado con GIST. Retorna BOOLEAN y puede usarse directamente desde la API antes de crear una cita.

SELECT verificar_disponibilidad('uuid-fisio', '2025-06-15 10:00', '2025-06-15 11:00');
Retorna TRUE si no hay conflicto, FALSE si ya hay una cita en ese rango (excluyendo estados cancelada/no_show).

8. Row Level Security (Multi-Tenancy)
PostgreSQL RLS garantiza que cada tenant solo pueda ver y modificar sus propios datos, incluso si comparten la misma base de datos física. La política se activa configurando la variable de sesión app.current_tenant_id al inicio de cada request HTTP.

8.1 Configuración por Request
-- En cada request, el middleware Node.js ejecuta:
SET app.current_tenant_id = '<uuid-del-tenant-del-usuario-autenticado>';

8.2 Tablas con RLS Activado
	•	tenants
	•	usuarios
	•	pacientes
	•	citas
	•	notas_sesion
	•	membresias
	•	pagos

🔒 Con RLS activo, un query como SELECT * FROM pacientes retorna únicamente los pacientes del tenant de la sesión actual, sin necesidad de añadir WHERE tenant_id = ? en cada consulta.

	•	API endpoints: autenticación, CRUD pacientes, agenda básica, membresías

	•	Trigger: progreso_dolor automático

	•	Módulos: ejercicios, ejercicios_asignados, ejercicios_registro

Fase 4 – Multi-tenant Completo (Semanas 10-12)
	•	Row Level Security en producción
	•	Panel SaaS: gestión de tenants, suscripciones Stripe
